# The reference graph runtime reads/writes the last-known-value lock-free and bounds in-process dispatch cycles with a depth cap

Status: accepted

> **Superseded in part by [RFC-0007](../spec/rfcs/0007-delivery-terminates-at-target.md) / [ADR-0051](0051-delivery-terminates-at-target-no-dispatch-limits.md)** (2026-07-04): the in-process dispatch-depth cap is deleted — SUBSCRIBER delivery now terminates at its target (no re-dispatch to the target's subscribers), so dispatch-level cycles are impossible by construction and no cap or termination machinery exists. This ADR's concurrency model (lock-free LKV, per-vertex atomicity, await) remains in force.

The L4 graph runtime (`core/` M3) is multi-threaded: publishers `write`, consumers `read` or block in `await`, concurrently, across threads. The protocol leaves the locking strategy implementation-defined ([ADR-0013](0013-v1-scope-boundaries.md)) and states only that **each vertex is the atomic unit** of a read/write ([02-graph-model.md](../reference/02-graph-model.md) §required atomic operations). This ADR records how the reference implementation realizes that, and how it terminates in-process subscriber cycles — both being load-bearing, hard-to-reverse choices that a reader would otherwise find surprising.

**The last-known-value slot is an atomic refcounted-pointer swap.** A vertex's value is held as `std::atomic<std::shared_ptr<const View>>`; `read` is an atomic load (returning a refcount-bump clone — no byte copy), `write` an atomic store. Neither takes the per-vertex mutex, so reads never block on writers and vice versa. The per-vertex mutex + `std::condition_variable` guard only the comparatively cold paths: the subscriber list, the stream history ring, settings, and the `await` waiter accounting (a write-sequence counter the condvar predicate watches, which is lost-wakeup-safe). This is the natural use of the atomic segment refcounts M2 already pays for ([ADR-0012](0012-modular-memory-binding-transparent-router.md), [02](../reference/02-graph-model.md) §required atomic operations: increment `relaxed`, decrement `acq_rel`).

> **"Lock-free" here means reads/writes do not take the vertex mutex — not wait-free hardware CAS.** libstdc++'s `std::atomic<std::shared_ptr<T>>::is_lock_free()` is `false`: it uses an internal sharded spinlock pool. The guarantee we rely on (and test under TSan) is *freedom from data races* and *reads not serializing behind the vertex lock*, not single-instruction atomicity. A future build MAY swap in a genuinely lock-free refcounted pointer (split reference count / hazard pointers) behind the same API. Under `LIBTRACER_NO_ATOMIC` the slot degrades to a plain pointer for single-threaded / Cortex-M0 targets.

**In-process dispatch cycles terminate by a depth cap, the analogue of the wire `hop_count`.** A `write` fans out to a vertex's subscribers; a spec-faithful SUBSCRIBER names a *target vertex*, so fan-out re-dispatches by writing to that vertex — which may itself have subscribers. A subscriber cycle (`/a → /b → /a`) would recurse forever. Re-dispatch therefore carries a depth counter and stops at `kMaxDispatchDepth` (32), dropping the over-deep dispatch. This is the in-process counterpart of the bridged-transport `hop_count`/`MAX_HOPS` guarantee ([ADR-0014](0014-router-cycle-termination-hop-count.md)): same constant, same role (a hard termination bound independent of any dedup), different layer — there is no ROUTER envelope or `origin_peer_id` in a single process, so depth is the only signal available.

## Considered options

- **A single coarse lock per vertex (mutex around read/write/await/subscribers).** Rejected: simplest, but reads serialize behind writers and behind each other, wasting the atomic refcounts M2 built and defeating the "as fast as zero-copy" claim ([02](../reference/02-graph-model.md) §same-substrate insight). The lock-free LKV is a small amount of extra care for a large hot-path win.
- **Single-threaded core (no internal locking; app serializes).** Rejected for the reference build: `await`'s contract is "block until the next write," which is only meaningful across threads; a single-threaded core degrades `await` to "return an already-buffered value." (`LIBTRACER_NO_ATOMIC` still offers this mode for bare-metal targets that own their serialization.)
- **Detect cycles by tracking visited vertices per dispatch instead of a depth cap.** Rejected for v1: a visited-set is allocation per dispatch and more code, for no better guarantee than a depth bound — the same trade ADR-0014 made for the wire path. The depth cap is O(1) and matches the wire model.

## Consequences

- `read`/`write` are wait-free of the vertex mutex; `await` and subscription mutation take it briefly. Fan-out snapshots the subscriber list under the lock, then dispatches outside it, so user callbacks (and recursive re-dispatch) never run under the vertex lock.
- In-process subscriber loops deliver at most `kMaxDispatchDepth` levels before the cap drops further re-dispatch — bounded duplicate delivery, never a hang or stack overflow. Matches the [ADR-0014](0014-router-cycle-termination-hop-count.md) trade for the wire path.
- The concurrency model is validated, not asserted: the M3 `graph_test` runs a multi-writer/multi-reader stress under **ThreadSanitizer** (no data races) and **ASan+UBSan** (no leaks/UB).
- This is an *implementation* decision about `core/`, not a wire/protocol commitment; per [ADR-0013](0013-v1-scope-boundaries.md) a different implementation MAY lock differently, provided it preserves per-vertex read/write atomicity and cycle termination. `kMaxDispatchDepth` reuses 32 to mirror the wire `MAX_HOPS`; the exact value is implementation-defined.
