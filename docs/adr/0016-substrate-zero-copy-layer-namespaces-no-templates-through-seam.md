# The L0/L1 memory substrate is a CPU-mediated zero-copy scatter-gather router: namespaces mirror the layer model and templates never cross the seam

Status: accepted

> **Superseded in part by [ADR-0047](0047-build-time-closed-module-sets-compile-time-seams.md)** (2026-07-04): §Decision 3's runtime-vtable dispatch and "templates never cross the seam" rule are replaced by build-time-closed module sets with tag dispatch. §Decisions 1–2 (CPU-mediated scatter-gather; namespaces mirror the layer model), the uniform-boundary-type rule, and the backend-private `alloc_hint_t` shape remain in force.

The memory substrate (`core/` M2 — `tr::mem` at L0, `tr::view` at L1) is the foundation every other layer stands on, and its reason to exist is a single load-bearing capability: **a source vertex's live memory — a user struct, an MMIO register, a DMA buffer — is the payload, gathered straight onto the wire and placed straight into the destination vertex's binding on a remote node, with no serialize step and no bounce buffer.** This is load-bearing claim 1 ("a TLV in memory IS the wire bytes") plus the transparent byte router ([ADR-0012](0012-modular-memory-binding-transparent-router.md)), stated as the substrate's *purpose* rather than a side effect. This ADR records the three hard-to-reverse, would-surprise-a-reader choices that realize it; the mechanical spellings (snake_case `_t`, `SCREAMING_SNAKE` enums, Doxygen) live in [`core/STYLE.md`](../../core/STYLE.md), not here.

## Decision 1 — transfer is CPU-mediated scatter-gather; assembly is rope-chaining, never `memcpy`

The substrate presents a logically-contiguous payload that physically lives in scattered backing memory as a **`rope_t`** — an ordered chain of `view_t` windows over (possibly different) `segment_t`s. **Assembly and reassembly mean constructing that rope by chaining views — pointer-linking, zero-copy.** They never mean copying scattered bytes into one contiguous buffer. The rope is also the **transport-agnostic scatter-gather representation**: each transport *lowers* it to whatever native DMA it has — `iovec`/`sendmsg` for IP sockets, descriptor chains for CAN DMA, verbs for an RDMA NIC. RDMA is one lowering among several, not a privileged path.

A contiguous **copy** occurs at exactly one place: a substrate boundary a transport's DMA cannot span (e.g. an lwIP `pbuf` rope forwarded to a CAN region), where the bridge/transport flattens at **egress**. Never per-fanout, never per-hop, never called "reassembly."

We adopt **CPU-mediated** transfer as the v1 contract (branch A), explicitly *not* true one-sided RDMA (branch B). The sender gathers the source binding via its transport's scatter-gather facility; the **remote CPU** validates the frame (CRC, strip header+trailer) and places the payload into the destination vertex's binding — zero-copy where the substrates are compatible, one bounded copy at a substrate boundary or when the destination is a register/FIFO. The trailer *forces* this: a wire frame is `header + payload + trailer(CRC)`, so a frame can never be raw-DMA'd into a destination register — only the payload belongs there, and only the CPU can strip and verify. True one-sided RDMA (remote NIC writes the destination with no remote-CPU involvement) is deferred to a later `transport_rdma`/`mem_rdma` pairing where, **when both peers load it, the same `write()` transparently lowers to a one-sided verb** — an implementation detail under the unchanged read/write/await API, not a wire or capability change.

## Decision 2 — namespaces mirror the layer model, making the L0↔L1 contract a greppable dependency rule

The C++ root namespace is `tr::`. Its sub-namespaces mirror the [six-layer model](../reference/00-overview.md#the-six-layer-model): **`tr::mem` = L0** (the substrate: `mem_backend_t` and the concrete backends, `io_dir_t`, `alloc_hint_t`); **`tr::view` = L1** (views and ownership: `segment_t`, `segment_ptr_t`, `view_t`, `rope_t`). `segment_t` lands in `tr::view`, not `tr::mem`, because the layer model assigns the *refcount* to L1 ("Views and ownership: … refcounted segments") while L0 owns only allocation/lifetime/cache — the `segment` is the object that crosses the seam (a backend *makes* it; the view layer *owns* it).

The payoff is enforcement, not aesthetics: a symbol in `tr::view` referencing `tr::mem` is a normal upward dependency, while a symbol in `tr::mem` reaching into `tr::view` is a **layering violation you can grep for**. The namespace boundary *is* the L0↔L1 contract.

`segment_t` is the one unavoidable complication, and we name it precisely. It is **mutually recursive** with `mem_backend_t` (the backend's `alloc` returns `segment_t*` and `destroy` takes one; the segment holds a `mem_backend_t*` back-pointer), so it is the **sanctioned L0↔L1 boundary type** — the single `tr::view` symbol the L0 `mem_backend_t` *interface* is permitted to name, because it is the contract object both layers are defined in terms of. Everything else stays strictly one-directional. Concretely:

- `alloc` returns a **raw `segment_t*`** (refcount = 1), never a `segment_ptr_t`. `segment_ptr_t` is pure L1 ownership machinery, *not* a boundary type, so the L0 interface must not name it; the caller adopts the raw pointer via `tr::view::segment_ptr_t::adopt`.
- Convenience helpers that *produce an owning handle* — `heap_alloc`, `borrow`, `borrow_const` (each returns a `segment_ptr_t`) — therefore live in **`tr::view`**, not `tr::mem`. They reference the backends upward (`mem::heap_backend()`, `mem::borrowed_backend()`). The pure-L0 `mem::heap_backend()` (returns `mem_backend_t&`) stays in `tr::mem`.

So the only sanctioned downward name is `view::segment_t` in the backend interface and its overrides; grep `tr::view` inside `tr::mem` and that boundary type is the sole legitimate hit.

Code sub-namespaces deliberately **avoid the eight error-concept words** (`frame`, `tlv`, `path`, `schema`, `flow`, `access`, `transport`, `version`) reserved by the `tr::` error-identity namespace ([ADR-0009](0009-built-in-error-model-tr-concept-namespace.md)). `tr::frame::*` is therefore always an error identity (a string path), never a C++ symbol; `tr::mem::pool_t` is always a C++ symbol. The two registers of `tr::` never collide.

## Decision 3 — templates are a type-safety tool *above* the seam, never a polymorphism mechanism *through* it

The substrate works because `segment_t` is **one uniform runtime type** carrying a `mem_backend_t*`. Aggressive templating is welcome where it **erases to identical codegen** — strong typedefs (`alloc_hint_t`, offsets, ids), `concept`-constrained signatures, `constexpr` layout, inlining CRTP — i.e. for "no boilerplate, no type violations." It is **forbidden** where it would monomorphize the ownership seam: templating `segment_t` on its backend type (`segment<Backend>`) would metastasize into `view<Backend>`, `rope<Backend>`, and the graph above them, exploding the binary and destroying the runtime-pluggable, heterogeneous-backend model that is the whole point of [ADR-0012](0012-modular-memory-binding-transparent-router.md).

The language floor is **C++23** — the standard the MCU toolchains actually implement (arm-none-eabi-gcc 14, ESP-IDF GCC, Clang 18). C++26 is used only **opportunistically behind `__cpp_*` feature-test macros with a C++23 fallback**; nothing in the MCU profile gates on `-std=c++26`. The MCU profile is `-fno-exceptions -fno-rtti -Os` with `std::expected`-based `Result<T>` and `LIBTRACER_NO_ATOMIC` for single-core targets. The enforcement that keeps "aggressive templating" honest is the existing **≤16 KB stripped Cortex-M0 sentinel test**: templating is permitted exactly as far as the sentinel stays green, turning the discipline into a CI invariant rather than a hope.

## Considered options

- **Branch B (true one-sided RDMA) as the v1 contract.** Rejected: needs per-vertex memory registration + rkey/remote-VA exchange — a handshake that directly contradicts the frozen "capability negotiation does not exist" decision ([ADR-0013](0013-v1-scope-boundaries.md), CONTEXT.md §Capability negotiation) and the aspirational `mem_rdma` backend. Branch A delivers the entire zero-copy vision under the unchanged API and leaves B available later as a transparent optimization.
- **One flat `tr::` namespace (or `tr::mem` covering both L0 and L1).** Rejected: erases in code the exact L0↔L1 boundary the hardening set out to make *clearer*; the layer line would survive only in comments, with no greppable enforcement.
- **`tr::frame` / `tr::flow` as code sub-namespaces (concept = module).** Rejected: re-introduces precisely the concept-vs-module conflation [ADR-0009](0009-built-in-error-model-tr-concept-namespace.md) forbids; a reader could no longer tell an error identity from a C++ symbol.
- **Static-polymorphism backends (CRTP/`concept` `segment<Backend>`), C++26 baseline.** Rejected: zero-cost virtual-call elision is not worth template-exploding `view`/`rope`/graph, losing heterogeneous runtime backends, and amputating the MCU target a C++26 requirement would cause. A host-class build tier *may* later require C++26, but only as a separately-fenced profile, never the core.
- **`alloc(size, uint32_t hint)` with a shared cross-backend hint vocabulary.** Rejected as a bloat vector: a shared hint registry invites every backend to grow interpretations of every flag. `alloc_hint_t` is instead an **opaque, backend-private strong typedef** — its meaning is private to the defining backend, no two backends share a value's meaning, and a hint-ignoring backend accepts any value.

## Consequences

- The substrate's seam is runtime-polymorphic and uniform: one `segment_t`, virtual `mem_backend_t`, one vtable — cheaper than N monomorphizations and compatible with the 16 KB budget. Pluggable/heterogeneous backends remain a `mem_backend_t*` indirection.
- Egress is true zero-copy where the transport offers scatter-gather (`rope_t::to_iovec` → `writev`/`sendmsg`); the single bridge-boundary flatten is the only copy, and it is named as such, never as "reassembly."
- The L0↔L1 dependency direction is mechanically checkable (grep for `tr::view` inside `tr::mem`). Higher layers qualify `view::view_t` / `mem::mem_backend_t`.
- This is an **implementation** decision about `core/`, not a wire/protocol commitment ([ADR-0013](0013-v1-scope-boundaries.md)): a conforming implementation in another language MAY structure its substrate differently, provided it preserves the wire format, the zero-copy semantics, and per-vertex read/write atomicity. The reference suite docs [09](../reference/09-memory-substrate.md) and [10](../reference/10-module-catalog.md) are updated to show this C++23 seam as the reference impl's, not a normative cross-language ABI.
- The cache-coherency hooks `before_io`/`after_io` carry transfer *timing*; `io_dir_t` carries *direction*; the backend maps the pair to clean/invalidate. They are no-ops on cacheless cores.
