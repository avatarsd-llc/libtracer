# The net side-channels (`bridge_t` *and* `fwd_router_t`'s children-table) dissolve into the vertex tree — transport-vertex as composite, connection-vertex as leaf, root as terminus resolver

Status: accepted. **Supersedes [ADR-0036](0036-bridge-dissolves-into-connection-vertex.md).** ADR-0036 was right that the net routing must dissolve into the vertex tree, but it named only `bridge_t` and said to "relocate its ROUTER logic verbatim." Grilling the actual code surfaced two corrections that retract that framing: the codebase carries *two* net-routing abstractions (not one), and the connection-vertex's `forward` is already a different, spec-normative model. This ADR broadens the scope to both side-channels and replaces "relocate verbatim" with a compositor split plus a hard lean contract. The *decision* (dissolve the net side-channels into the path tree per [ADR-0027](0027-transport-and-connections-are-vertices.md)) stands; the *mechanism* is corrected here.

[ADR-0027](0027-transport-and-connections-are-vertices.md) makes every transport and each connection a first-class `/` vertex — "one path tree," data endpoints and transports uniformly. The C++ core landed *two* pre-0027 side-channels that each duplicate what the path tree now owns:

- **`bridge_t`** (M4, [ADR-0014](0014-router-cycle-termination-hop-count.md)) — a standalone class over one `transport_t&` with `export_vertex` (egress: subscribe a local vertex, ROUTER-wrap each write, send) and a mount-point ingress (unwrap → dedup `(origin, ts)` → `hop_count`/`MAX_HOPS` → `graph.write(mount)`). Flood/mirror routing; the ROUTER envelope is the cycle-termination vehicle.
- **`fwd_router_t`** (RFC-0004 / [ADR-0035](0035-implementing-rfc-0004-remote-operation-addressing.md) slice 3) — a stateless hop-by-hop FWD forwarder holding a `vector<child_t{name, transport_t*}>` plus `child_by_segment()`. Source-routed: resolve the first `dst` segment, strip it, forward the suffix, grow `src` (the way back); at a terminus, `op_resolver_t` applies READ/WRITE/AWAIT and returns `FWD{REPLY}`.

ADR-0036 named only `bridge_t`. But `fwd_router_t`'s `children_` table is *isomorphic* to a transport-vertex with connection-vertex children (`{name:"ws", link:&ws}` is exactly `/net/ws` resolved by NAME = path segment) — the **same disease** as `bridge_t`'s `export_vertex`: a pre-0027 side-channel duplicating the path tree. And `fwd_router_t::route_fwd`'s forward branch already *is* "strip my segment, send the suffix" — the very behavior ADR-0036 assigned to the connection-vertex's `forward`. So the connection-vertex's routing is `fwd_router_t`'s model, not `bridge_t`'s; "relocate `bridge_t` verbatim" was wrong.

## Decision

**Dissolve both side-channels into the vertex tree via a compositor over the vertex interface.** A vertex is either a leaf connection or a composite of connections; the caller cannot tell them apart (the compositor invariant). Three roles:

- **Transport-vertex = composite.** Groups connections via `:children[]` ([ADR-0017](0017-in-band-vertex-creation-controller-orchestration.md) / [ADR-0027](0027-transport-and-connections-are-vertices.md)). Forwards by resolving the next path segment to a child — which is `graph.find(child)` against the vertex map that already exists. **No new class, no resident routing state, no children-table.** This replaces `fwd_router_t::children_` + `child_by_segment`.
- **Connection-vertex = leaf.** Holds one `transport_t&` (its link). `forward` = strip its own segment, send the unresolved suffix across the link (zero-copy scatter-gather — the relocated `route_fwd` forward logic). `receive` = the ingress guard (dedup `(origin, ts)` + `hop_count`/`MAX_HOPS`, absorbed from `bridge_t` as a per-link safety layer) + the optional label plane (ADVERTISE/COMPACT/NACK, per-link) + proxy-mount write (`/net/<conn>/<remote path>` stores the latest inbound value, fan-outs to the one subscribed input — *one pub/sub*, per ADR-0036). This replaces `bridge_t` (ingress half) and `fwd_router_t`'s forward/label halves.
- **Root = terminus resolver.** The one non-vertex piece: `op_resolver_t` (READ/WRITE/AWAIT → `FWD{REPLY}` back along `src`). A terminus is "this node is the destination" — node-level, not per-link. It is the node's *self*.

**The connection-vertex carries the FWD source-routing model, not the ROUTER model.** `bridge_t::export_vertex` + ROUTER-wrap egress is **deleted** — source-routing has no "subscribe-and-mirror-to-one-transport"; you write `FWD{WRITE, dst=/net/<conn>/<path>, payload}` and the connection-vertex forwards it. `bridge_t`'s ingress dedup/`hop_count` guard is **absorbed** (not relocated) as a per-connection defense-in-depth layer: a looping or malicious `dst` still needs a bound, but source-routing doesn't flood, so the recent-set is a guard, not the routing mechanism.

**Endpoint duality — and the leanness mechanism.** A transport-vertex is an *addressing* endpoint always (path-addressable: `:settings`, `await` link up/down, `:children[]`, readable) whether it fronts one link or many. It is **not** a *data* endpoint — no value storage, no subscriber list, no fan-out state; it is a pure name→child demux. The connection-vertex (leaf) *is* a data endpoint via its proxy mount. The composite's statelessness **is** the leanness mechanism: the grouping layer carries zero data state, so it is the cheapest possible vertex.

### The lean contract (hard invariants, not aspirations)

The router must be as lean as the smallest node demands. Five invariants gate the dissolved design:

1. **Forward hops never full-decode.** Dispatch on the top-level type byte + the first ~3 headers read by offset (type, op, first `dst` segment — the entire forward decision); a full `wire::decode` (recursive, `vector<tlv_t>` children) runs **only at the terminus**, where the resolver must read the tree. Today `on_frame` full-decodes every inbound frame and the forward branch then ignores the tree — pure overhead, and on a 16KB node a feasibility failure.
2. **Zero heap on the forward path.** The fresh header bytes (`head1` = FWD-hdr + op + shrunk-dst-hdr; `head2` = grown-src-hdr + prepended inbound NAME, each ≤ ~32B) are emitted into **pooled, refcounted `segment_ptr_t`**s from the M2 `mem_pool_t` (fixed-capacity, statically-allocated arena — no `malloc`, no fragmentation), roped in and sent via `to_iovec()`. The `iov` (≤6 spans) is a stack `std::array`, not a `std::vector`. No `std::vector` byte-traffic per hop.
3. **Per-connection state is single-writer.** The dedup ring and the label table live in the connection-vertex, touched only by that connection's one receive thread — lock-free (fixed-size ring / open-addressed set) or at worst a cheap spinlock with **no cross-link contention**. The node-global `std::mutex` + three `std::map` of today's `route_handle_t` dies.
4. **The composite carries no data state** (above): transport-vertex is pure demux; value/subscriber/fan-out live at the leaf and root.
5. **Terminus decode is arena-backed.** `wire::decode` is extended with a decode-into-arena variant (borrowed/pool-backed `tlv_t` tree, not `vector<tlv_t>`) so the whole node hot path — forward *and* terminus — is heap-free.

### The 16KB-RAM feasibility gate

A minimal node has ~16KB of RAM. That is too tight for heap reliance on the hot path (fragmentation kills the node before latency does) and too tight for lwIP (its PCB + pbuf footprint alone consumes the budget). So the 16KB node runs a **synchronous substrate** (CAN / UART / a lean driver) where `send` returns after the bytes are in the TX mailbox — the borrowed inbound `span` and the pooled head segments are consumed before return, so the forward path is heap-free *and* zero-copy-to-the-wire with no async-lifetime problem.

**Stage-2 retirement of `bridge_t` and `fwd_router_t` is gated on a 16KB-RAM build** (CAN transport, no heap, no lwIP) that forwards a frame with **zero heap allocations and zero heap growth**, *measured* (an allocator counter / `__wrap_malloc`), not asserted. If it cannot, the design is wrong, not the node. Pool capacity and the decode-arena size are per-node `:settings` tunables on the transport-vertex — which is what `:settings` is for.

### Scoped out (M5+, not a blocker)

**The lwIP async-refcount receive-seam is deferred.** True zero-copy on an *async* substrate (UDP-over-lwIP `PBUF_REF`, RDMA) requires the receive path to hand up a refcounted `segment_ptr_t` (not a raw `span`) so the rope can hold a ref across the async send. That is a real `transport_t::receiver_t` seam upgrade, scoped to the M5+ real-socket transports — it does **not** block the 16KB node, whose synchronous drivers never need it. **WS-over-lwIP is acknowledged one-copy** (TCP must hold bytes for retransmit until ACK; the IDF WS layer copies) — not zero-copy, not pretended to be. UDP-over-lwIP zero-copy depends on the deferred seam.

## Staged (carried by [#83](https://github.com/avatarsd-llc/libtracer/issues/83), sub-issue of [#59](https://github.com/avatarsd-llc/libtracer/issues/59))

1. **Stage 1 — shell over the live path.** Transports/connections appear as `/` vertices (`:settings`, `await`, `:children[]`-created) while `bridge_t`/`fwd_router_t` still carry the bytes. Proves the vertex/compositor model with zero regression to the TSan-clean path. Prerequisite: the `:children[]` SPEC creation sub-issue [#82](https://github.com/avatarsd-llc/libtracer/issues/82).
2. **Stage 2 — the flip, gated on the 16KB-zero-heap bench.** Split `fwd_router_t` into transport-vertex demux / connection-vertex forward+receive+labels / root resolver; relocate the forward offset-scan and the terminus resolver; absorb the ingress guard per-connection; delete `bridge_t::export_vertex` + ROUTER egress and `fwd_router_t::children_`; land the five lean invariants. Public API change → `core/CHANGELOG.md` note. Retire `bridge_t` and `fwd_router_t` as user-facing classes.

## Considered options

- **Amend ADR-0036 in place.** Rejected: the original's "relocate `bridge_t` verbatim" is *retracted* (the ROUTER egress is deleted, not relocated) and the scope doubled (`fwd_router_t` too). A retraction is what the repo's convention records as a supersede ([ADR-0009](0009-built-in-error-model-tr-concept-namespace.md), [ADR-0028](0028-native-cores-kept-consistent-by-conformance-vectors.md)) — preserving the first-cut record, not silently rewriting a merged ADR.
- **Keep `bridge_t`'s ROUTER model as the connection-vertex routing; retire `fwd_router_t`.** Rejected: ROUTER-flood-with-dedup contradicts the stateless, source-routed, normative model (RFC-0004 / [ADR-0035](0035-implementing-rfc-0004-remote-operation-addressing.md)); [reference/04 §bridge](../reference/04-communication-flows.md) ("indistinguishable from local-source data") is `fwd_router_t`'s `deliver_local`-as-write, not a flood. `fwd_router_t` is the model to keep; `bridge_t`'s egress is the part that dies.
- **Relocate `fwd_router_t` verbatim, keeping the per-frame `wire::decode` and the `std::vector` heap allocs.** Rejected: violates the lean contract. On a 16KB-RAM node the per-frame decode-alloc and the three per-hop vector-allocs are a *feasibility* failure, not a perf nicety. "Relocate verbatim, don't rewrite" survives only for the offset-scan and the terminus resolver; the forward *entry* gets the no-decode/no-alloc treatment, justified by the bench.
- **Per-subscriber egress proxy vertex.** Rejected (carried from ADR-0036): [ADR-0021](0021-colon-field-plane-is-the-vertex-ioctl.md) forbids turning `:subscribers` into `/` sub-vertices. The route is the path, not a proxy vertex.

## Consequences

- **Both** `bridge_t` and `fwd_router_t` dissolve (not just `bridge_t`). `fwd_router_t`'s 420-line monolith splits into three small, individually-TSan-clean units (transport-vertex demux / connection-vertex forward+receive+labels / root resolver).
- The node hot path is **heap-free by construction**; pool exhaustion is **backpressure (drop / await), not OOM**. 16KB-RAM minimal nodes are feasible as forwarders and sensors.
- Public API removal: `export_vertex`/`set_mount` (from ADR-0036) **and** `fwd_router_t`'s `add_child`/`advertise`/`send_compact` surface — replaced by `:children[]` writes and ordinary `FWD{WRITE}`. `core/CHANGELOG.md` note. The `tr::net` namespace stays; the two classes go.
- The lwIP async-refcount receive-seam is a deferred M5+ task; UDP-over-lwIP zero-copy depends on it; WS-over-lwIP stays one-copy. The 16KB node is unaffected (synchronous substrate).
- This reopens [#86](https://github.com/avatarsd-llc/libtracer/issues/86) (reference-vs-code divergence) honestly: `wire::decode`-into-`vector` and `fwd_router`-into-`vector` violate invariants #5 and #2 — they are part of the divergence to fix as Stage-2 prereqs, not just the dissolve.
