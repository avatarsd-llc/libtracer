# The net-plane performance model: two-plane forwarding (stateless full-route vs. label-compacted), a per-connection pooled substrate, and a typed buffer-lifetime seam — with two corrections to ADR-0037

Status: accepted. **Refined by [ADR-0039](0039-pmr-memory-model-host-aligned-allocation.md)** (the PMR memory model: "zero-heap" means the *steady-state* forward hop — init/setup/terminus/host allocations are out of scope — and every non-hot-path allocation draws from an injected `std::pmr::memory_resource`, unifying with the L0 `mem_backend_t` seam; invariants #2/#5 sharpened accordingly). **Refines [ADR-0037](0037-net-side-channels-dissolve-into-vertex-tree-compositor.md)** (the compositor dissolution) and **corrects two of its claims** against [RFC-0004](../spec/rfcs/0004-remote-operation-addressing.md) and the actual code. ADR-0037 fixed the *structure* (transport-vertex = composite, connection-vertex = leaf, root = resolver) and asserted five lean invariants. This ADR fixes the *dynamics* — what the forward path does per hop, where state lives, how buffers are owned — and repairs the two invariant claims that do not survive contact with the wire format and the threading model. The compositor decision stands; the guard model and the single-writer claim are corrected here.

## Context

ADR-0037's lean contract is right in spirit (heap-free hot path, single-writer per-connection state, forward hops never full-decode) but two of its sentences are wrong in fact, and a grill across the *zero-copy / lock-free / latency / reliability* axes surfaced both plus the model that actually satisfies them.

- **Wrong claim 1 — "absorb the `bridge_t` ingress guard (dedup `(origin, ts)` + `hop_count`/`MAX_HOPS`) per-connection" ([ADR-0037](0037-net-side-channels-dissolve-into-vertex-tree-compositor.md) §Decision).** A bare `FWD` frame carries **no** `origin`, `ts`, or `hop_count` field — [RFC-0004](../spec/rfcs/0004-remote-operation-addressing.md) §D is normative: *"`FWD` is loop-free by construction … It carries **no** `origin`/`hop_count`/dedup — those stay in `ROUTER` on the multi-path delivery side."* The code agrees: `fwd_router.cpp` has zero dedup/`hop_count`/recent-set machinery. So there is no per-connection guard to "absorb" onto the FWD path; the guard ADR-0037 names is `bridge_t`'s ROUTER-envelope machinery, which is `std::mutex`-guarded and heap-allocates a 24-byte key **per frame** (`recent_key()`), directly violating ADR-0037's own invariants #2 (zero-heap forward) and #4 (composite carries no data state). The wire-format contradiction and the lock-free/zero-copy contradiction are the same defect seen twice.
- **Wrong claim 2 — invariant #3: "the dedup ring and the label table … touched only by that connection's one receive thread — lock-free."** The label table's **egress** half (`route_handle_t::egress_`, `egress_label_`, `next_label_`) is mutated by `ensure_egress`, which `deliver_remote` calls from the **producer fan-out thread** (`set_remote_delivery_sink`), *and* by `on_nack` from the receive thread — two writers, possibly N producer threads on a shared link. That is exactly why `route_handle.hpp:158` is a node-global `std::mutex` today, not a design accident. "One receive thread, lock-free" is false for egress.

The grill also examined a tempting simplification the maintainer proposed — **one immutable full path + a single advancing cursor**, where each hop does `cursor += 1` and the bytes never move — and, after a follow-up, **rejected it for the general wire**. It **cannot carry the return route** under local link-naming: the pre-cursor segments were written in *other nodes'* namespaces, so an intermediate relay cannot read its own reverse-link name out of them. Reply routing then needs either stored per-request correlation state (the locked, stateful thing being deleted) or a separately-accumulated `src` — and deferred producer fan-out (a delivery seconds after subscribe, no live socket) has no cursor-back path at all. Worse, keeping the path immutable means **retransmitting the consumed prefix every hop**, so it is heavier than the shrinking-`dst` model *and* still needs `src` — no net win. What is genuinely kept is only ordinary **zero-copy view-parsing of the remaining `dst`** (already in `route_fwd`), which is not the cursor. The single-cursor round trip earns its keep **only** under **symmetric link-naming** (where the immutable path *is* the round trip and `src` vanishes) — an RFC-0004 amendment that forecloses unidirectional CAN. See §Considered options.

## Decision

**Adopt a two-plane forward model over a per-connection pooled-segment substrate, and use each mechanism only where its cost is free.** The four axes genuinely conflict — the leanest mechanism (a `u16` label) is the least reliable-by-construction (soft state, self-heal), and the most reliable (stateless full-route) is the fattest on the wire. There is no single mechanism that maxes all four; the design is which mechanism carries which flow.

### 1. Two forward planes, split by flow character

- **One-shots and cold flows → stateless full-route** ([RFC-0004](../spec/rfcs/0004-remote-operation-addressing.md) §B/§D). The remaining `dst` (forward route) is read **zero-copy as a view** into the inbound buffer — the consumed leading segment simply is not carried onward, so `dst` **shrinks** on the wire. `src` (return route) grows by a **zero-copy rope head-prepend** of this node's *local* name for the inbound link. Route lives in the frame ⇒ the forwarder is **stateless**, the reply self-routes home, and a hop may reboot mid-operation. This is the reliable plane; it pays the bytes because a one-shot pays them once. (This is *not* the immutable-path-**cursor** — see §Considered options: under local naming that model retransmits the consumed prefix every hop and still needs `src`, so it is strictly heavier; it earns its keep only in the symmetric-naming profile.)
- **Established streams → label-compacted** ([RFC-0004](../spec/rfcs/0004-remote-operation-addressing.md) §E.1, `route_handle_t`). A per-link `u16` label aliases the established route; the hot delivery is a `COMPACT{label, payload}` — a **direct array index**, no NAME scan, no route bytes. This is the lean/fast plane; it can't afford full-route bytes (1 kHz × ~60 B route on a 4 B sample ≈ 16×, and impossible in CAN's 8 bytes) and it *can* amortize the one-time advertise.
- **Lazy self-heal bridges them:** first delivery advertises a label, thereafter `COMPACT`; `clear_link` on reconnect drops the binding so the next delivery re-advertises; a delivery on a stale label is `HANDLE_NACK`'d and relearned, never misrouted. No transport "up" event required.

**One addressing model, not two** (pinning the [ADR-0036](0036-bridge-dissolves-into-connection-vertex.md) proxy-mount ↔ FWD-terminus question). ADR-0036 kept a receive-side *proxy mount* — inbound data lands at `/net/<conn>/<remote path>`, `graph.write(mount)` → fan-out — while [RFC-0004](../spec/rfcs/0004-remote-operation-addressing.md) §D routes a remote delivery as a `FWD{WRITE}` to a `dst` the terminus resolves via `deliver_local`. These are **the same operation, not two**: when a `FWD{WRITE}` whose `dst` has been fully consumed arrives at a connection-vertex, the terminus write it performs **is** the proxy-mount write — `/net/<conn>/<remote path>` is exactly the local name for "the fully-shifted `dst` at this leaf." Per [CONTEXT.md §Path-as-route](../../CONTEXT.md), the send-side suffix a caller routes *through* a transport vertex equals the receive-side prefix that vertex mounts inbound data *under* — so the proxy mount is the terminus `deliver_local`, addressed from the leaf's local root. There is no separate "bridge republish" step and no second addressing scheme; ADR-0036's proxy mount is the Stage-2 name for the FWD terminus write, and the ROUTER-envelope republish it inherited from `bridge_t` is the deleted egress (per ADR-0037).

The maintainer's single-immutable-path intuition is therefore **not discarded** — for the general (locally-named) topology it *is* the label-compacted frame (the whole route collapsed to a `u16`); it becomes the literal base frame only under an optional symmetric-naming profile (below).

### 2. The forward hop — offset-dispatch, zero-heap, single owner

- **Never full-decode** (invariant #1, kept). Dispatch on the top-level type byte + the first ~3 headers read by offset (type, op, first `dst` segment); the recursive `wire::decode` into `vector<tlv_t>` runs **only at the terminus**. Today's `on_frame` decodes every frame and the forward branch discards the tree — the single largest waste on the path, and a 16KB-node feasibility failure.
- **Zero heap** (invariant #2, kept, mechanism pinned). The inbound frame is **never touched**; `dst`/`src`/`payload` are **views** into it. The only fresh bytes per hop are the small synthesized heads (FWD header + op + the prepended inbound `NAME`, ≤ ~32 B), emitted into **pooled `segment_ptr_t`** from `mem_pool_t`. The egress `iov` is a **stack `std::array<span, ≤8>`**, not a `std::vector`. Pool exhaustion → **backpressure (drop / await), never `malloc`, never OOM.**
- **`src` is a separately-accumulated region, not folded into `dst`.** `dst` is the shrinking forward-remaining view; `src` is the growing return-accumulated rope. They cannot be one immutable sequence read by a single cursor because forward-names and reverse-names are physically different symbols for different links at each node (the reason the immutable-path-cursor fails under local naming — §Considered options).

### 3. State ownership — corrected from invariant #3

Invariant #3 is **restated honestly**, split by responsibility (SRP: learning an inbound route and advertising an outbound route are two responsibilities, two writers, two lifetimes):

- **No `(origin, ts)` dedup state on the connection at all.** It is stateful and only meaningful once a delivery is `ROUTER`-wrapped for a cyclic/multi-path region ([RFC-0004](../spec/rfcs/0004-remote-operation-addressing.md) §D) — so it lives **only on the ROUTER-wrapped delivery sink**, paid only where cycles exist. The bare-FWD loop bound is **`dst`-monotonicity** (`dst` shrinks one segment per hop ⇒ termination in ≤ len(`dst`) hops, itself MTU-bounded) plus a **stateless per-frame segment cap** (refuse a `dst` deeper than N, N from `MAX_HOPS`) — a counter read, no set, no lock, no heap.
- **Ingress label table → receive-thread-owned, genuinely lock-free single-writer.** (This half of invariant #3 was always true.)
- **Egress label table → per-connection, hot path lock-free / flow-setup spinlocked.** `next_label_` becomes one `std::atomic<u16>` fetch-add. The per-delivery `COMPACT` emit is a lock-free array read of an already-bound label. The *cold* advertise-once path (fan-out thread × NACK-re-advertise) takes a **per-connection spinlock with no cross-link contention** — precisely invariant #3's own sanctioned fallback (*"or at worst a cheap spinlock"*). The honest claim is **"hot path lock-free, flow-setup per-connection-spinlocked,"** not "all lock-free." The node-global `std::mutex` + four `std::map` of today's `route_handle_t` dies; each connection owns its slice as a **fixed-capacity open-addressed table** (`u16` → slot, routes in pooled segments), sized by `:settings`; exhaustion falls back to full-route `FWD{WRITE}`.

### 3b. The `children_` dissolution is a `tr::net` registry, not `graph.find` — and it is a separate brick from the ROUTER retirement

[ADR-0037](0037-net-side-channels-dissolve-into-vertex-tree-compositor.md) §Decision phrases the transport-vertex demux as *"`graph.find(child)` against the vertex map that already exists"* and §Stage-2 bundles *"delete `bridge_t::export_vertex` + ROUTER egress **and** `fwd_router_t::children_`"* as one step. Implementing it surfaced two corrections:

1. **`graph.find` cannot return a transport without inverting the layers.** `graph_t::find` returns an L4 `vertex_t`, and `vertex.hpp` is explicit that this L4 struct *"never depends on `tr::net`"* — a vertex must not carry a `transport_t*` ([ADR-0016](0016-substrate-zero-copy-layer-namespaces-no-templates-through-seam.md)). So the demux cannot literally be `graph.find(child) → transport`. The **layering-safe realization** (Brick 3a, landed) is a single `tr::net`-owned `child_registry_t` (the `NAME → transport*` table + `by_segment`/`by_name`) that `fwd_router_t` holds and `transport_vertex_t` populates — **one owner, no duplicated table**, achieving ADR-0037's "no children-table duplication" goal without an L4→L5 dependency. The registry is `tr::net`, exactly where knowing a transport is allowed. The eventual "a `/net/<conn>` path resolves to its connection" addressing still routes *through* the vertex tree; it just resolves the *link* via the L5 registry keyed by the same NAME, not by storing a transport in an L4 vertex.
2. **The `children_` dissolution and the `bridge_t`/ROUTER retirement are different bricks.** [RFC-0004](../spec/rfcs/0004-remote-operation-addressing.md) keeps `ROUTER` *"unchanged … on the cyclic/multi-path delivery side"* — and `bridge_t` **is** the ROUTER-plane implementation. So `bridge_t` cannot be deleted as a side effect of unifying the FWD registry; the cyclic-delivery mechanism it carries needs a replacement home first. Brick 3a unifies the registry (FWD plane) with zero behavior change; the ROUTER-plane migration + `bridge_t` retirement is a **later, separate brick** with its own design (where the `(origin, ts)` dedup + `hop_count` guard live once `bridge_t` dissolves — the per-connection ingress-guard of ADR-0037 §Decision, but only for ROUTER-wrapped cyclic deliveries, per §1 above).

### 3a. The intra-device path pays nothing (hard invariant)

**A write between two endpoints on the *same device* must be no slower than a direct function call + pointer dereference** — the net plane adds zero cost to a local edge. This is load-bearing: a device is *one path tree*, and the compositor exists to make remote *look* local, never to make local *pay* remote. Concretely, in the reference impl the fan-out dispatch (`graph_t::fan_out`) branches per subscriber: a **local** edge takes the direct `callback(view)` leg (a `std::function` call over a refcount-bumped view, no byte copy) or the `find(target_key)` + `write_impl` leg (a map lookup + call — the "deref"); the transport/`remote_sink_` leg is entered **only** when the subscriber carries a non-empty `link`. A local subscriber never has a `link`, so it never touches the router, the label plane, `wire::encode`, or any `:children[]`/`/net` machinery — those live in a disjoint subtree. **The transport-vertex/connection machinery is off the local hot path by construction, not by optimization.** Measured: introducing `transport_vertex_t` (#83 Stage-1) left the in-process `inproc`/`inproc-borrow` perf-gate points unchanged (p50 flat vs. baseline). Stage-2 MUST preserve this — the compositor's `find(child)` demux is reached only for a `dst` that names a transport child; a local `dst` resolves to a local vertex on the first hop and never enters the net plane.

### 4. Buffer lifetime — a typed seam, synchronous-first

Three lifetime classes, one contract: **the inbound backing is inviolate; only pooled heads are fresh; the rope holds the refcount across the send.**

- **Inbound frame.** On a **synchronous** substrate (CAN / UART — the 16KB node) it is a *borrowed span* valid for the receive callback; `send()` returns after the bytes are in the TX mailbox, so the borrow is released before the callback exits — **zero refcount, zero copy, TSan-clean by construction.** On an **async** substrate (WS / UDP / lwIP) the buffer may free before the queued send drains, so ingress must hand up a **refcounted `segment_ptr_t`** the egress rope pins until send completes.
- **The `transport_t::receiver_t` seam is typed as a refcounted `segment_ptr_t` now, and the synchronous release path is implemented first.** A synchronous transport releases immediately (refcount is a no-op cost); an async transport pins. This makes the 16KB CAN node and the WS node run the **identical forward path** — the seam's *shape* is what is expensive to change later, so it is fixed now even though async pinning (`PBUF_REF` zero-copy) lands in M5+. Until async pinning lands, async transports do a **one-copy at ingress** (copy into a pooled segment) — slower but safe, never a use-after-free. **WS-over-lwIP stays one-copy regardless** (TCP holds bytes for retransmit).

### Reliability ordering (why the split is not a compromise)

Most-to-least reliable-by-construction: **stateless full-route** (survives a hop reboot; no per-hop state to corrupt) > **label-compacted** (soft per-link state; needs NACK + re-advertise self-heal) > **symmetric-naming immutable-round-trip** (needs a link-naming handshake; cannot express unidirectional CAN). The two-plane split assigns the reliable mechanism to the flows that want reliability (one-shots) and the lean mechanism to the flows that can amortize its self-heal (streams) — each cost paid only where it is free.

## Considered options

- **One immutable full path + a single advancing cursor for *both* `dst` and `src`** (the maintainer's proposal). **Not adopted for the general (locally-named) wire** — a follow-up grill showed it earns nothing there and costs bytes. Two distinct things were briefly conflated: (a) reading the remaining `dst` **zero-copy as a view** — real, correct, and *already in the code* (`route_fwd`), but that is ordinary zero-copy parsing, not the cursor; (b) transmitting the **full path immutable + a wire cursor** so nothing shrinks — the actual proposal. Under local naming (b) is strictly heavier: it **retransmits the consumed prefix every hop** (≈16 B path re-sent at hop 3 vs. a shrunk ≈4 B) *and still requires* `src` to grow (pre-cursor segments are in other nodes' namespaces, so a relay cannot route a reply from them without stored correlation state, and deferred fan-out has no cursor-back path). The one saving it claims — avoiding a `dst` header rebuild — is already free: a fresh outer `FWD` header is synthesized into a pooled head every hop regardless (its length changed because `src` grew), and the 4-byte `dst` header rides in that same pooled segment. So the cursor is **redundant with zero-copy view-parsing and pessimal on wire bytes** here. It is **not** an independent optimization; it is the *defining mechanism* of the symmetric-naming profile (next option) and earns its keep **only** there — where the immutable path *is* the round trip (cursor forward, then backward for the reply) and `src` disappears entirely.
- **Symmetric link-naming profile** (both endpoints agree on a link's name ⇒ the one immutable path routes the round trip, `src` vanishes, the leanest possible frame). Recorded as a **legitimate optional profile** for a strictly-bidirectional lean tree (e.g. web-UI ↔ robot ↔ *bridged* CAN), **not** the general wire: it needs a name-agreement handshake at every link, cannot express a unidirectional link (receive-only CAN, broadcast radio) or an asymmetric return route, and lightly erodes the per-device-own-root property. Would be an RFC-0004 amendment, deferred; the general wire stays local-named with `src` compacting to a label for streams.
- **Keep dedup `(origin, ts)` on the connection as a per-link guard** (ADR-0037 as written). Rejected: bare FWD carries no such fields, it heap-allocates per frame and needs a lock — violating invariants #2/#3/#4 — and source-routing does not flood, so there is nothing to dedup until a delivery is ROUTER-wrapped. Guard belongs on the ROUTER sink, paid only where cycles exist.
- **Serialize producer fan-out per connection** so egress is truly single-writer and even the spinlock drops. Rejected as the default: it taxes the common hot case (steady deliveries) to protect the rare cold one (N producers establishing new compact flows on one shared link simultaneously). Kept as an option for a **single-producer-single-link leaf**, where it is strictly simpler and the serialization point never contends.
- **Untyped `span` receiver seam, one-copy async forever.** Rejected as the seam shape: ships faster but leaves async permanently one-copy and forces a breaking `receiver_t` change to ever reach `PBUF_REF` zero-copy. Type it as `segment_ptr_t` now, implement sync-release first.

## Consequences

- **ADR-0037 invariant #3 is superseded by §3 here** ("hot path lock-free, flow-setup per-connection-spinlocked"; no dedup state on the connection). ADR-0037's "absorb the `hop_count`/dedup guard per-connection" sentence is **retracted** — replaced by `dst`-monotonicity + a stateless segment cap on FWD, and `(origin, ts)` dedup confined to the ROUTER-wrapped delivery sink. Invariants #1, #2, #4, #5 stand.
- **The Stage-2 flip ([#83](https://github.com/avatarsd-llc/libtracer/issues/83)) inherits a sharper contract:** offset-dispatch forward; `dst` shrinking zero-copy view + `src` pooled rope-prepend; per-connection open-addressed label table (ingress lock-free, egress hot-lock-free/cold-spinlocked); the typed `segment_ptr_t` receiver seam (sync release first). The 16KB-RAM zero-heap bench gate is unchanged and now has a concrete target: the synchronous CAN forward path allocating **zero** heap, measured.
- **Sequencing is proven-ness-ordered:** ship the synchronous 16KB path first (borrow-and-release, no refcount seam exercised, zero-heap measurable, TSan-clean by construction); gate async zero-copy behind the typed seam (M5+, the lwIP `PBUF_REF` work ADR-0037 already scoped out).
- **No wire change, no spec change.** FWD/FIELD/ROUTER/ADVERTISE/COMPACT/HANDLE_NACK are unchanged; this is an L4/`tr::net` implementation-shape decision, like [ADR-0035](0035-implementing-rfc-0004-remote-operation-addressing.md). The symmetric-naming profile, *if* ever pursued, is the one piece that would need an RFC-0004 amendment.
- **Feeds the [#86](https://github.com/avatarsd-llc/libtracer/issues/86) reference sweep:** `route_handle_t`'s node-global mutex and the per-frame decode are named here as the concrete divergences to retire, alongside the ROUTER-bridge prose still current in reference 00/03/04/07/10/11.
