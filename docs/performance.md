# Performance

> Honest, reproducible numbers against [Eclipse Zenoh](https://zenoh.io), swept
> across a parameter grid (payload size × subscriber fan-out × topic count) so the
> comparison is a **response surface**, not a single point. Full harness,
> methodology, and the raw CSV live in
> [`bench/`](https://github.com/avatarsd-llc/libtracer/tree/main/bench); this page
> is the summary. Speed-ups are machine-independent; absolute rates are
> representative single runs (see [`bench/RESULTS.md`](https://github.com/avatarsd-llc/libtracer/blob/main/bench/RESULTS.md)).

**Bottom line: libtracer beats zenoh-c on both throughput and latency on every
path** — in-process, network latency, and (via scatter-gather composition) network
throughput. `rmw_tracer` and `rmw_zenoh` are thin wrappers over these engines, so
this engine delta is the dominant term in the ROS-level result.

## The deep comparison (system dynamics)

libtracer's throughput advantage over zenoh-c across the payload × fan-out grid —
it starts at ~1.5× (single subscriber) and **widens to ~6.7× as fan-out grows**,
because libtracer's per-delivery dispatch overhead is far below zenoh's per-sample
machinery:

![libtracer/zenoh throughput speedup across payload × fan-out](https://raw.githubusercontent.com/avatarsd-llc/libtracer/main/bench/figures/speedup_throughput_fanout.png)

The same advantage in latency (zenoh p50 ÷ libtracer p50), the throughput response
surfaces, and the per-axis line plots are generated by `bench/grid.sh` into
[`bench/figures/`](https://github.com/avatarsd-llc/libtracer/tree/main/bench/figures):
`speedup_latency_fanout.png`, `surface_throughput_fanout.png`,
`throughput_vs_fanout.png`, `latency_vs_fanout.png`, and the topic-count
equivalents (`*_endpoints.png`).

## In-process — the zero-copy story

One process; a publish is a refcount bump on a shared `segment_ptr_t` — the same TLV
bytes are the wire encoding *and* the in-memory value *and* the graph node, so the
hand-off moves **zero bytes**. Across the matrix:

| axis | libtracer vs zenoh |
| --- | --- |
| fan-out 1 → 8192 subscribers | **2.5× → 6.4×** throughput, **2.6× → 6.6×** lower p50 |
| payload 1 B → 8 KB (1 sub) | **2.2× → 2.6×** throughput/latency |
| topics 1 → 8192 (1 sub) | **1.5× → 1.7×** throughput, lower latency |

The zero-copy `inproc-borrow` path (the loaned-message path, [ADR-0025](https://github.com/avatarsd-llc/libtracer/blob/main/docs/adr/0025-rmw-tracer-end-to-end-zero-copy-rcl-over-rdma.md))
is **flat ~80 ns p50 even at 8 KB** (zenoh ~320 ns) — payload size doesn't move it,
because nothing is copied.

## Network — latency *and* throughput (real UDP, two processes)

Two processes over localhost UDP; one-way latency via `CLOCK_MONOTONIC`.

- **Latency: libtracer wins ~4.5×** — ~14 µs p50 vs zenoh ~64 µs, every size. Its
  per-message path is thin (encode a VALUE TLV → ROUTER-wrap → one `sendto`), with
  nothing between the write and the wire.
- **Throughput: libtracer wins via scatter-gather composition.** A composite
  endpoint's value is a **rope already batched in memory**; the transport ships it
  with **one `sendmsg(iovec)`** ([`transport_t::send(iov)`](https://github.com/avatarsd-llc/libtracer/blob/main/core/include/libtracer/transport.hpp)),
  so one syscall carries K values. Effective throughput scales with composition
  size K **at flat latency** — 5.1M values/s @ ~3 µs (K=8) up to 46.6M @ ~12 µs
  (K=256), versus zenoh-c's 3.5M @ 62 µs.

```{note}
This is the key architectural difference. Zenoh raises throughput with a **timer
batcher** — which is *why* its latency is ~4.5× worse (every message waits for the
batch). libtracer batches **structurally**: the composition *is* the batch, shipped
zero-copy via `sendmsg`/`writev`/RDMA-SGE — no Nagle timer, so no latency penalty.
You get both. (One `sendto`-per-message also remains available for the
lowest-possible single-message latency.)
```

## Reproduce it

```sh
git clone https://github.com/avatarsd-llc/libtracer
cd libtracer/bench
./fetch_zenoh.sh     # vendors prebuilt zenoh-c (x86_64 linux)
./run.sh             # in-process sweep (fan-out / payload / topics / mixed)
./run_net.sh         # network (two processes over UDP)
python3 -m venv .venv && ./.venv/bin/pip install matplotlib numpy
./grid.sh            # response-surface grid + 2D/3D figures into figures/
```

Full tables, methodology, and the bit-level rationale are in
[`bench/README.md`](https://github.com/avatarsd-llc/libtracer/blob/main/bench/README.md)
and [`bench/RESULTS.md`](https://github.com/avatarsd-llc/libtracer/blob/main/bench/RESULTS.md).
